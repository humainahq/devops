Section 01

Purpose

This SOP defines a disciplined workflow for teams working on the same repository. It applies to all developers contributing to shared projects.

Branch strategy at a glance: main is production. dev is shared integration. feature/* branches are developer-specific work.

Goals of this workflow:

  • Avoid merge conflicts and broken integrations
  • Maintain a stable, always-working integration branch
  • Enable parallel development without blocking each other
Section 02

Core Principles

Rule 1  —  No direct commits to main or dev. Everything goes through Pull Requests (PRs).
Rule 2  —  Developers work in isolated feature branches.
Rule 3  —  dev is always in a working, deployable state.

Section 04

Developer Workflow

1
Sync dev
2
Create branch
3
Work & push
4
Stay updated
5
Open PR
6
Delete branch

1 Sync with latest dev

Always pull the latest state of dev before you start any new work.

bash
git checkout dev
git pull origin dev

2 Create feature branch

Name branches with the feature/ prefix followed by a short, descriptive name.

bash
git checkout -b feature/<name>

# Example
git checkout -b feature/login-api

3 Work and push

Make your changes, commit clearly, and push your feature branch to origin.

bash
git add .
git commit -m "Add: login API"
git push origin feature/login-api

4 Keep branch updated (Critical in teams)

While you're working, teammates will merge into dev. You must sync your branch periodically to stay up to date and avoid large conflicts later.

bash
git fetch origin
git merge origin/dev

# Or use rebase if your team allows it
# git rebase origin/dev
Tip: Merge dev into your feature branch often — small syncs are much easier than one massive conflict resolution session.

5 Create PR (feature → dev)

  • Open a Pull Request targeting dev — never main
  • Request a review from at least one teammate
  • Resolve any conflicts that arise
  • Merge only after review approval

6 Delete branch after merge

Once the PR is merged, clean up both the remote and local feature branches.

bash
# Delete remote branch
git push origin --delete feature/login-api

# Delete local branch
git branch -d feature/login-api
Section 05

Release Workflow (dev → main)

When dev is stable and tested, cut a release to production via PR.

dev (stable)
→
PR: dev → main
→
Review & approve
→
Merge to main
→
Deploy
Never skip review on the dev → main PR. This is your last line of defense before production.
Section 06

Handling Multiple Developers

Two developers working simultaneously on different features is the standard scenario. Here is exactly how it plays out:

Developer A — feature/login

git checkout -b feature/login-api ... works, commits ... git push origin feature/login-api Opens PR → dev. Merges first.

Developer B — feature/dashboard

git checkout -b feature/dashboard ... works, commits ... git push origin feature/dashboard Must sync before their PR.

After Developer A merges, dev has changed. Developer B must update before opening their PR:

bash — Developer B syncs after A's merge
git checkout feature/dashboard
git fetch origin
git merge origin/dev

# Resolve any conflicts, then push
git push origin feature/dashboard

# Now open the PR
Section 07

Conflict Handling

Conflicts happen when:

  • Two developers modify the same lines in the same file
  • Your branch has fallen behind dev for too long

Resolution process:

bash — Conflict resolution steps
# 1. Fetch and merge the latest dev into your branch
git fetch origin
git merge origin/dev

# 2. Git will mark conflicting files. Open them and resolve.
#    Look for conflict markers:  <<<<<<<  =======  >>>>>>>

# 3. After resolving, stage and commit
git add .
git commit -m "Merge: resolve conflicts with dev"

# 4. Push updated branch
git push origin feature/your-branch
Prevention is better than resolution. Sync with dev every day or two if your branch lives longer than 24 hours.
Section 08

Rules for the Team

✓  Mandatory

✅ Always pull latest dev before branching
✅ Always update your branch before raising a PR
✅ Always use PR reviews — no self-merges
✅ Delete feature branches after merge

✗  Forbidden

❌ Direct push to main
❌ Direct push to dev
❌ Long-running feature branches (max 2–3 days)
❌ Merging without resolving conflicts
Section 09

Hotfix Workflow (Production Issues)

A hotfix is an emergency fix directly branched from main. It bypasses dev for speed, but must be backported to dev immediately after.

bash — Hotfix process
# 1. Branch from main (not dev)
git checkout main
git pull origin main
git checkout -b hotfix/critical-fix

# 2. Apply fix, commit, push
git add .
git commit -m "Hotfix: fix critical login issue"
git push origin hotfix/critical-fix

Then raise two PRs:

hotfix/critical-fix
→
PR → main
(deploy fix)
+
PR → dev
(backport fix)
Critical: Never skip the PR back to dev. If you don't backport, the fix will be lost the next time dev is released to production.
Section 10

Sync main → dev (When Needed)

Only do this in two situations: after a hotfix is applied directly to main, or if any direct change was made to main that dev doesn't have.

bash
git checkout dev
git merge origin/main
git push origin dev
Section 11

Best Practices

Practice Why it matters
Keep PRs small and focused Easier to review, less risk of conflicts, faster to merge
Use clear commit messages Future-you and teammates will thank you. Use prefix conventions: Add: Fix: Refactor:
Merge dev into your branch frequently Small incremental syncs prevent large, painful conflict sessions
Communicate if touching shared files Tell teammates before editing config files, shared utilities, or database schemas
Don't let branches live too long Feature branches older than 2–3 days diverge quickly and become hard to merge
Section 12

Example Flow

A complete end-to-end example with two developers working in parallel:

Dev A — Auth Feature

git checkout dev git pull git checkout -b feature/auth ... code, commit ... git push origin feature/auth PR: feature/auth → dev Review → Merge ✓

Dev B — UI Feature

git checkout dev git pull git checkout -b feature/ui ... code, commit ... git fetch && git merge origin/dev git push origin feature/ui PR: feature/ui → dev Review → Merge ✓

Once both features are in dev and tested:

bash — Release to production
# Open PR: dev → main, review, merge
# Deploy main to production
Section 13

Summary

Branch Purpose Who merges here Direct push?
main Production code PR from dev or hotfix/* Never
dev Shared integration PR from feature/* Never
feature/* Developer work in isolation Developer directly Yes (your branch)
hotfix/* Emergency production fixes Developer directly Yes (your branch)
Core takeaway: dev is the central working branch. Every developer uses isolated feature branches. Sync frequently to avoid conflicts. All merges go through PR review. This workflow scales cleanly for any team size.
